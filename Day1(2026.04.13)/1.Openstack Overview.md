# Chapter 1. Openstack Overview

## 1-1. OpenStack이란?

서버 자원을 제공하기 위한 오픈소스 클라우드 인프라 플랫폼.
주로 VM(가상머신) 워크로드를 생성하고 관리한다.

서버 자원을 제공하는 방식은 크게 두 가지가 있다.
- **VM 방식**: Host OS 위에서 실행되는 Hypervisor가 Guest OS까지 독립적으로 생성하여 격리된 환경을 제공
- **Container 방식**: Host OS를 공유하는 형태로 경량화된 환경을 제공

VM 워크로드를 관리하는 Hypervisor의 종류로는 KVM, ESXi(VMware), Citrix Hypervisor(XenServer), Hyper-V(Microsoft) 등이 있다.

---

## OpenStack 구성요소 — TripleO 구조

**TripleO(OpenStack On OpenStack)**: Undercloud(관리용 OpenStack) 위에서 Overcloud(운영용 OpenStack)를 배포·관리하는 구조.

---

### 1. Undercloud (관리 환경)

Overcloud 환경의 노드를 프로비저닝, 배포, 관리하는 역할을 담당.

| 구성요소 | 역할 |
|---|---|
| **Nova** | VM 배포 및 관리 |
| **Neutron** | 네트워크 관리 |
| **Cinder** | Block Storage |
| **Swift** | Object Storage |
| **Keystone** | 인증 처리 |

---

### 2. Overcloud (운영/Production 환경)

실제 IaaS 자원을 사용자에게 제공하는 환경.

#### Compute Node
- Hypervisor(주로 KVM)가 존재하며, VM이 실제로 배포되는 노드
- Nova(컴퓨팅), Neutron(네트워킹) 에이전트가 동작

#### Storage Node
- Block Storage, Object Storage, File System을 모두 지원하는 **Ceph**(분산 스토리지)를 주로 사용
- 필요에 따라 다양한 스토리지 백엔드 구성 가능

#### Network Node
- Neutron을 구성할 수 있으나, 최근에는 주로 **Compute Node에서 네트워크를 직접 관리**하는 방식(DVR, Distributed Virtual Router)을 사용

#### Controller Node
- 전체 클러스터를 관리하는 중앙 제어 노드
- Nova API, Neutron Server, Keystone, Horizon(대시보드) 등 핵심 서비스가 동작
- 이 노드가 다운되면 전체 환경에 영향을 주므로, **보통 3개 노드로 HA(High Availability) 구성**

## OpenStack 구성요소 — 리소스 구성

### 계층 구조
Cloud > Region > Domain > Project > AZ (순으로 범위가 좁아짐)

---

### Cloud
OpenStack 배포 전체를 지칭하는 단위.
컴퓨팅(서버), 스토리지, 네트워크 등 IaaS 자원을 제공하는 전체 환경.

---

### Region
하나 이상의 데이터센터로 구성되는 지리적 단위.
- 각 Region은 **독립적인 API 엔드포인트**를 가지며, 서비스 간 완전한 독립성 보장
- Keystone(인증)만 Region 간 공유하고, 나머지 서비스(Nova, Neutron 등)는 Region별로 독립 운영
- 예: `kr-seoul-1`, `us-east-1`

---

### Domain
Keystone의 **최상위 관리 단위**. 사용자·사용자그룹·프로젝트를 논리적으로 묶은 단위.
Domain 단위로 인증 서비스 진행됨.
- 기업/조직 단위로 분리하여 관리할 때 사용
- Domain별로 관리자(Domain Admin)를 지정해 권한을 위임할 수 있음
- 기본값: `Default` 도메인

---

### Project (= Tenant)
실제 자원을 사용하고 과금되는 **논리적 격리 단위**.
- 사용자가 VM, Volume, 네트워크 자원을 할당받아 사용하는 공간
- **Quota(할당량)** 를 통해 프로젝트별 자원 사용량 상한선 설정 가능
  - 예: vCPU 20개, RAM 50GB, Instance 10개 등
- 사용자는 여러 프로젝트에 속할 수 있으며, 프로젝트별로 다른 역할(Role) 부여 가능

---

### Availability Zone (AZ)
Region 내에서 **물리적으로 격리된 위치** 단위.
- Fault Domain(동일 장애에 영향받는 범위) 기준으로 구성 — 전원, 네트워크, 랙 등을 분리
- VM을 여러 AZ에 분산 배치하여 단일 장애점(SPOF) 방지
- **Nova AZ**: 컴퓨팅 자원(VM) 배치 단위
- **Cinder AZ**: 스토리지 자원(Volume) 배치 단위 (별도 구성)
- 내부적으로 **Host Aggregate**(물리 호스트 그룹)와 연동되어 특정 하드웨어 그룹에 VM을 배치하는 데 활용

## IaaS 위에 올라가는 VM 구성요소

VM 생성 시 가장 핵심은 Guest OS 설치이며, 이를 위해 아래 요소들이 필요하다.

| 구성요소 | 설명 | 관리 서비스 | 필수 여부 |
|---|---|---|---|
| **OS Image** | Guest OS 설치에 사용되는 이미지 (예: Ubuntu, CentOS) | Glance | ✅ 필수 |
| **Flavor** | VM의 하드웨어 스펙 정의 (vCPU 수, RAM 크기, Disk 크기) | Nova | ✅ 필수 |
| **Network** | VM에 연결되는 가상 네트워크 및 IP | Neutron | ⚙️ 선택 (생성 후 attach 가능) |

---

## VM 디스크 구성 — Ephemeral vs Persistent

### Ephemeral Disk (임시 디스크) — 기본값
- Glance의 OS Image로부터 생성되는 **임시 디스크**
- **VM과 생명주기를 같이함** — VM 삭제 시 디스크 데이터도 함께 삭제됨
- Stateless 워크로드(무상태)에 적합

### Persistent Disk (영구 디스크) — Stateful 저장
VM과 독립적으로 데이터를 영구 보존하고 싶을 때 사용. 종류는 아래와 같다.

#### 1. Volume (Block Storage) — Cinder
- Block Storage 형태의 논리 단위 (Logical Unit)
- VM에 디스크처럼 attach/detach 가능
- **Boot from Volume**: OS 자체를 Volume에 설치하여 VM 삭제 후에도 OS 상태 보존 가능
- 주로 DB, 고성능 I/O가 필요한 워크로드에 적합

#### 2. Shared Filesystem (NAS) — Manila
- NFS/CIFS 기반의 **공유 파일시스템**
- 여러 VM이 동시에 같은 스토리지를 마운트하여 사용 가능
- 주로 다중 VM 간 파일 공유가 필요한 워크로드에 적합

#### 3. Object Storage — Swift (or S3 호환)
- Key-Value 형태로 데이터를 저장하는 **오브젝트 스토리지**
- 파일을 직접 마운트하지 않고 API(HTTP)로 접근
- 대용량 비정형 데이터(이미지, 백업, 로그) 저장에 적합
